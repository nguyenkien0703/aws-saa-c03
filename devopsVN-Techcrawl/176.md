![image](https://images.spr.so/cdn-cgi/imagedelivery/j42No7y-dcokJuNgXeA0ig/d22c45ab-6ee7-491f-b574-38e479f5a491/Untitled/w=1920,quality=90,fit=scale-down)

Thay vì để chúng gọi nhau như bình thường thì ta sẽ nhúng Istio Sidecar Proxy (Data Plane) vào hai Services này để chúng gọi nhau thông qua Proxy, lúc này thì toàn bộ Request ra hoặc vào của hai Services đều được theo dõi bởi Istio Sidecar Proxy.

![image](https://images.spr.so/cdn-cgi/imagedelivery/j42No7y-dcokJuNgXeA0ig/a661a58f-4d1d-4abf-8537-cbfc0459d2ce/Untitled/w=1920,quality=90,fit=scale-down)

## Chuẩn bị

Ta tải source code ở đây [Istio In Action](https://github.com/hoalongnatsu/istio-in-action), toàn bộ ví dụ trong series này đều nằm trong source code trên.

Tiếp theo, ta tạo một `namespace` tên là `istio-action` cho toàn bộ ví dụ trong series.

Copy

```
kubectl create ns istio-action
```

Chuyển `namespace` trên thành `namespace` mặc định.

Copy

```
kubectl config set-context $(kubectl config current-context) --namespace=istio-action
```

## Triển khai Catalog Service

Đầu tiên ta sẽ tạo Catalog Service, xem thử cấu hình của Catalog Service.

Copy

```
cat services/catalog/kubernetes/catalog.yaml
```

Copy

Trước khi tạo Catalog Service ta sẽ nhúng thêm đoạn cấu hình của Istio Sidecar Proxy vào, có hai cách để làm như sau:

- Dùng câu lệnh `istioctl kube-inject`
- Bật tự động nhúng

## Nhúng Sidecar Proxy thủ công

Để nhúng Sidecar Proxy thủ công ta sẽ dùng câu lệnh `istioctl`, thực hiện như sau.

Copy

```
istioctl kube-inject -f services/catalog/kubernetes/catalog.yaml
```

Ta sẽ thấy nó in ra kết quả rất dài, `istioctl` sẽ thêm vào tệp tin cấu hình ban đầu thuộc tính `initContainers` và một Container tên là `istio-proxy`.

Copy

> Lưu ý: để cấu hình được Istio Sidecar Proxy thì Kubernetes Cluster phải có quyền NET\_ADMIN và NET\_RAW

Ta có thể dùng `kube-inject` để sửa tệp tin Deployment như trên và triển khai nó lên Kubernetes, hoặc bật tự động nhúng như ở phần tiếp theo, mình gợi ý là các bạn nên làm theo cách thứ hai.

## Nhúng Sidecar Proxy tự động

Để bật tự động nhúng cho Istio rất đơn giản, ta chỉ cần đánh label tên là `istio-injection=enabled` vào `namespace` mà ta cần.

Ví dụ ở bài này ta đang làm trên `namespace` tên là `istio-action`.

Copy

```
kubectl label namespace istio-action istio-injection=enabled
```

Bây giờ khi ta tạo Deployment thì Istio sẽ tự động nhúng Sidecar Proxy vào.

Copy

```
kubectl apply -f services/catalog/kubernetes/catalog.yaml
```

Copy

```
serviceaccount/catalog created
service/catalog created
deployment.apps/catalog created
```

Kiểm tra.

Copy

```
kubectl get pod
```

Copy

```
NAME                       READY   STATUS            RESTARTS   AGE
catalog-5d8f997684-fjvf5   0/2     PodInitializing   0          4s
```

Nếu `STATUS` chưa ở trạng thái `Running` thì chờ một chút và gọi lại.

Copy

```
kubectl get pod
```

Copy

```
NAME                       READY   STATUS    RESTARTS   AGE
catalog-5d8f997684-fjvf5   2/2     Running   0          61s
```

Nếu thấy `STATUS` ở trạng thái `Running` thì tất cả đã chạy xong, lúc này ta sẽ thấy ở trường `READY` là `2/2` tương ứng với hai Container, một Container là ứng dụng Catalog của ta và Container còn lại là `istio-proxy` Sidecar.

![image](https://images.spr.so/cdn-cgi/imagedelivery/j42No7y-dcokJuNgXeA0ig/1699c849-868f-408e-b5bd-ee5c0558e70f/Untitled/w=1920,quality=90,fit=scale-down)

Kiểm tra thử Catalog Service của ta đang chạy với Istio Proxy thì có xử lý được Request hay không.

Copy

```

```

## Triển khai Webapp Service

Tiếp theo ta tạo Webapp Service, xem thử cấu hình của Webapp Service.

Copy

```
cat services/webapp/kubernetes/webapp.yaml
```

Copy

Triển khai

Copy

```
kubectl apply -f services/webapp/kubernetes/webapp.yaml
```

Đợi khi nó tạo xong và gọi thử vào Webapp Service.

Copy

```
kubectl run -i -n default --rm --restart=Never dummy --image=curlimages/curl --command -- sh -c 'curl -s http://webapp.istio-action/api/catalog/items/1'
```

Khi ta gọi vào Webapp Service thì nó sẽ gửi Request qua Catalog Service để lấy dữ liệu, để dễ hình dung thì các bạn truy cập vào Webapp Service ở trên trình duyệt, chạy câu lệnh `port-forward`.

Copy

```
kubectl port-forward deploy/webapp 8080:8080
```

Mở trình duyệt và truy cập vào `localhost:8080`, ta sẽ thấy giao diện như bên dưới.

![image](https://images.spr.so/cdn-cgi/imagedelivery/j42No7y-dcokJuNgXeA0ig/669529c7-75e1-4097-9833-ae9aaec2813c/Untitled/w=1920,quality=90,fit=scale-down)

Bây giờ thì tất cả các Request vào và ra khỏi Webapp Service và Catalog Service đều đi qua Istio Proxy.

![image](https://images.spr.so/cdn-cgi/imagedelivery/j42No7y-dcokJuNgXeA0ig/32d5102e-08ab-46d5-b2b8-ad0dfd0c1a54/Untitled/w=1920,quality=90,fit=scale-down)

Như ta đã nói ở bài đầu tiên thì việc này sẽ giúp Istio có thể kiểm soát và theo dõi được toàn bộ Request trong hệ thống, đây là tính năng của cái gọi là Service Mesh mà người ta vẫn hay nói.
