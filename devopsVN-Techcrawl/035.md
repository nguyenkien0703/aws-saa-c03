## Thiết kế

Kết thúc bài trước hạ tầng của ta như sau:

![image](https://images.spr.so/cdn-cgi/imagedelivery/j42No7y-dcokJuNgXeA0ig/11b9871f-c375-4456-bc61-c007c63319db/aws-cdk-02-01.drawio/w=1920,quality=90,fit=scale-down)

Hiện tại để lấy danh sách câu hỏi và câu trả lời người dùng sẽ gọi tới máy chủ (EC2) và EC2 sẽ truy vấn trực tiếp vào Database (RDS). Thông thường khi người dùng tạo câu hỏi hoặc câu trả lời thì họ rất ít khi thay đổi. Nên truy vấn liên tục vào RDS để lấy cùng một kết quả dẫn tới việc hao phí tài nguyên. Bên cạnh đó truy vấn kết quả trực tiếp từ RDS khá chậm.

![image](https://images.spr.so/cdn-cgi/imagedelivery/j42No7y-dcokJuNgXeA0ig/4a3b14c0-43d0-44c9-aba7-05a1fc0a1e7e/aws-cdk-03-01.drawio/w=1920,quality=90,fit=scale-down)

Nên ta thêm một tầng cache ở giữa EC2 và RDS để tăng tốc độ đọc dữ liệu và giảm tải cho RDS. AWS cung cấp cho ta AWS Elasticache để làm cache.

![image](https://images.spr.so/cdn-cgi/imagedelivery/j42No7y-dcokJuNgXeA0ig/5aec3699-7917-41d0-b0f5-36123727c95e/aws-cdk-03-02.drawio/w=1920,quality=90,fit=scale-down)

Ngoài việc làm cache thì Elasticache còn giúp ta tăng độ khả dụng (High Availability) cho ứng dụng. Ví dụ nếu con RDS có chết thì người dùng tuy không thể ghi dữ liệu nhưng vẫn có thể đọc dữ liệu.

![image](https://images.spr.so/cdn-cgi/imagedelivery/j42No7y-dcokJuNgXeA0ig/8a906882-2e4c-4e73-90aa-91cd3d1e477f/aws-cdk-03-03.drawio/w=1920,quality=90,fit=scale-down)

Tuy nhiên nếu con EC2 của ta chết thì người dùng không thể truy cập ứng dụng được. Nên để tăng độ khả dụng ta cần tách riêng con EC2 cho việc đọc và ghi dữ liệu.

![image](https://images.spr.so/cdn-cgi/imagedelivery/j42No7y-dcokJuNgXeA0ig/d99582f8-f8b6-4dd7-b70c-1b59dcd786fd/aws-cdk-03-04.drawio/w=1920,quality=90,fit=scale-down)

Hệ thống hiện tại vẻ ổn nhưng nó có một điểm yếu là nếu người dùng ghi liên tục vào RDS ở một thời điểm, và vì dữ liệu đọc của ta ghi ở Elasticache nên ta sẽ thực hiện cập nhật Elasticache liên tục. Tuy nhiên với ứng dụng Q&A việc cập nhật dữ liệu theo thời gian thực như vậy không cần thiết => dẫn tới việc lãng phí tài nguyên của Elasticache.

Ta có thể thực hiện cập nhật dữ liệu đọc của Elasticache trong khoảng thời gian 5 phút một lần, toàn bộ dữ liệu được ghi vào RDS trong vòng 5 phút sẽ được cập nhật một lúc. Điều này giúp ta tránh lãng phí tài nguyên của Elasticache.

![image](https://images.spr.so/cdn-cgi/imagedelivery/j42No7y-dcokJuNgXeA0ig/11da6aed-25d7-42b8-84ba-4ce932d98c69/aws-cdk-03-05.drawio/w=1920,quality=90,fit=scale-down)

Thay vì phải truy vấn RDS và đọc dữ liệu trong vòng 5 phút. Ta có thể thực hiện như sau để đơn giản hóa công việc này:

1. Lưu dữ liệu vào RDS
2. Lấy ID của dữ liệu đó lưu vào một Database tạm khác

Để cho các bạn biết cách tạo thêm nhiều dịch vụ của AWS bằng CDK thì Database tạm mình dùng DynamoDB.

![image](https://images.spr.so/cdn-cgi/imagedelivery/j42No7y-dcokJuNgXeA0ig/2527889f-cecf-4346-9674-fa5b6837bb46/aws-cdk-03-06.drawio/w=1920,quality=90,fit=scale-down)

Để dễ dàng cho việc mở rộng và quản lí về sau, ta chia từng mục liên quan thành một Stack trong CDK.

![image](https://images.spr.so/cdn-cgi/imagedelivery/j42No7y-dcokJuNgXeA0ig/bcc9a257-ee64-4f33-830f-2d2f9b6f0450/aws-cdk-02-03.drawio/w=1920,quality=90,fit=scale-down)

## Chuẩn bị

Tạo thư mục và khởi tạo ứng dụng:

Copy

```
mkdir question-service && cd question-service
```

Copy

```
cdk init app --language go
```

Copy

```
go get
```

Xóa hết code trong tệp tin `question-service.go` và dán đoạn code sau vào:

Copy

Tạo một thư mục tên là `stack`:

Copy

```
mkdir stack
```

Tạo 3 tệp tin sau trong thư mục `stack`:

Copy

```
├── cache-stack.go
├── insert-stack.go
└── worker-stack.go
```

Cấu trúc thư mục hiện tại:

Copy

```
...
├── go.mod
├── go.sum
├── question-service.go
├── question-service_test.go
└── stack
    ├── cache-stack.go
    ├── insert-stack.go
    └── worker-stack.go
```

Từng tệp tin trong thư mục `stack` sẽ tương ứng với một Stack trong CDK. Ta dán đoạn code sau vào từng tệp tin.

Copy

## Viết code

**QuestionInsertStack**

Bắt đầu với `QuestionInsertStack` vì đây là Stack ta đã tạo ở bài trước.

![image](https://images.spr.so/cdn-cgi/imagedelivery/j42No7y-dcokJuNgXeA0ig/3a07c5cb-dfe7-49c2-9545-61ded1de97d0/aws-cdk-02-02.drawio/w=1920,quality=90,fit=scale-down)

Mở tệp tin `insert-stack.go` và dán đoạn code sau vào:

Copy

Tạo CloudFormation để xem trước các tài nguyên được tạo. Với một ứng dụng có nhiều Stack thì ta thêm tên Stack vào sau câu lệnh `synth`:

Copy

```
cdk synth QuestionInsertStack
```

**QuestionCacheStack**

Tiếp theo ta làm việc với `QuestionCacheStack`.

![image](https://images.spr.so/cdn-cgi/imagedelivery/j42No7y-dcokJuNgXeA0ig/2c4f27b0-37c4-470a-9e0e-44238389fe97/aws-cdk-03-08.drawio/w=1920,quality=90,fit=scale-down)

`QuestionCacheStack` bao gồm ba Construct là VPC, EC2, Elasticache. Với VPC, EC2 là Construct ta khá quen thuộc. Mở tệp tin `cache-stack.go` và dán đoạn code để tạo VPC và EC2 Construct trước:

Copy

Sau đó để tạo Elasticache Construct ta dùng hàm `NewCfnCacheCluster`:

Copy

Ta tạo Elasticache với loại là Redis và số lượng Node là 1.

Khác với EC2 Construct *(L2 Construct)* thì Elasticache Construct là *L1 Construct*. Khi ta tạo nó sẽ không có sẵn Security Group để cho phép tài nguyên khác truy cập Port của nó (mình sẽ giải thích khái niệm **L Construct** ở bài sau). Nên ta cần tạo Security Group cho Elasticache:

Copy

Ta tạo Security Group cho phép truy cập Port 6379 và gán nó vào Elasticache. Tạo CloudFormation cho `QuestionCacheStack`:

Copy

```
cdk synth QuestionCacheStack
```

**QuestionWorkerStack**

Cuối cùng ta làm việc với `QuestionWorkerStack`.

![image](https://images.spr.so/cdn-cgi/imagedelivery/j42No7y-dcokJuNgXeA0ig/22273691-f1e3-4353-9b1f-c50171895c96/aws-cdk-03-09.drawio/w=1920,quality=90,fit=scale-down)

Copy

## Triển khai

Để triển khai ứng dụng với nhiều Stack ta cần thêm tên Stack vào câu lệnh `deploy`. Ta triển từng Stack theo thứ tự:

QuestionCacheStack:

Copy

```
cdk deploy QuestionCacheStack
```

Copy


QuestionInsertStack:

Copy

```

```

Kiểm tra AWS Console ta sẽ thấy các tài nguyên của ta. Sau khi làm xong nhớ xóa tài nguyên để tránh mất tiền, chạy câu lệnh `destroy` với tên Stack theo thứ tự:

Copy

```
cdk destroy QuestionCacheStack
```

Copy

```
cdk deploy QuestionWorkerStack
```

Copy

```
cdk deploy QuestionInsertStack
```
